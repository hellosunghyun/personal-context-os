# 단점과 트레이드오프

## 여기서 볼 것

이 방식의 단점과 오용 가능성을 알고, 충분히 의식한 상태에서 사용합니다.

---

## 1. 컨텍스트가 길어질 수 있다

많이 넣을수록 좋아질 것 같지만 항상 그렇지 않습니다. 긴 문서는 오래된 정보, 덜 중요한 정보, 현재와 맞지 않는 정보를 함께 담습니다.

출처 메모: 긴 컨텍스트가 항상 좋은 결과로 이어지지 않는다는 관점은 [context engineering](../references.md#anthropic-엔지니어링-문서)과 긴 작업 하니스 문서의 배경을 참고했습니다.

### 대응

```text
전체 사용 설명서는 원본으로 보관한다.
실제 대화에는 Active Context를 주로 첨부한다.
Active Context는 1~3페이지로 유지한다.
```

---

## 2. 오래된 자기 설명이 현재를 가릴 수 있다

예전에는 맞았지만 지금은 아닌 내용이 계속 답변에 영향을 줄 수 있습니다.

### 대응

```text
삭제하지 말고 archive로 옮긴다.
“과거에는 맞았지만 현재는 약해짐”이라고 표시한다.
월 1회 Active Context를 갱신한다.
```

---

## 3. GPT가 너무 업그레이드해줄 수 있다

문장은 좋아졌는데 사용자 말처럼 느껴지지 않는 경우가 있습니다. 이 레포에서 원하는 것은 최고의 카피가 아니라 **제가 납득하고 수정 없이 쓸 수 있는 답변**입니다.

### 대응

```text
제가 요청하기 전 얼핏 상상한 방향과 맞는지 확인한다.
멋있지만 낯설면 낮춘다.
제가 쓰지 않을 표현은 제거한다.
```

---

## 4. 만족스러운 답변이 정확한 답변은 아닐 수 있다

마음에 드는 답변이 항상 정확한 답변은 아닙니다. 자기합리화가 될 수도 있습니다.

### 대응

```text
반대 근거를 요청한다.
다른 사람이 볼 관점을 요청한다.
검증 가능한 사실과 해석을 분리한다.
평가할 수 없는 내용은 평가 불가로 남긴다.
```

정량 점수로 완전히 평가하기 어렵다면, 좋은 답변과 나쁜 답변의 실제 예시만 남겨도 됩니다. 반복해서 보이는 차이가 생기면 그때 Active Context에 반영합니다.

출처 메모: 이 문서의 품질 확인 방식은 학술적 평가라기보다 실사용 회귀 테스트에 가깝습니다. 평가 항목을 고정하고 반복 비교하는 관점은 [OpenAI 평가 가이드](../references.md#openai-공식-문서)와 [Anthropic evals 문서](../references.md#anthropic-엔지니어링-문서)를 참고했습니다.

---

## 5. 팀원 설명서가 라벨링 도구가 될 수 있다

상대의 설명서를 보고 “이 사람은 원래 이래”라고 단정하면 위험합니다.

### 대응

```text
설명서는 직접 대화 전의 보조 도구로만 쓴다.
상대의 상태를 단정하지 않는다.
“제가 이렇게 이해했는데 맞나요?”라고 확인한다.
```

---

## 6. 직접 대화를 피하는 핑계가 될 수 있다

LLM에게 물어보는 것이 더 편해서 실제 대화를 미룰 수 있습니다.

### 대응

```text
갈등 해결의 최종 단계는 직접 대화다.
LLM은 질문 준비와 맥락 정리에만 쓴다.
```

---

## 7. 보안 리스크가 있다

회사 자료, 인사평가, 심리/건강 정보는 조심해야 합니다.

### 대응

```text
가명화한다.
요약본만 넣는다.
공개 범위를 표시한다.
민감 원문은 넣지 않는다.
```

출처 메모: 민감 정보와 비식별화의 한계는 [NIST 비식별화 보고서](../references.md#보안-개인정보-비식별화)와 [개인정보보호위원회 생성형 AI 안내서](../references.md#보안-개인정보-비식별화)를 참고하세요.
