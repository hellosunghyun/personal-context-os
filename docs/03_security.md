# 보안과 경계

## 여기서 볼 것

어떤 자료를 넣어도 되는지, 어떤 자료는 가명화해야 하는지, 어떤 자료는 넣지 말아야 하는지 정합니다.

---

## 먼저 결론

개인 맥락을 다루기 때문에 보안 기준을 먼저 정해야 합니다.

```text
개인 이야기라도 회사와 타인의 정보가 섞이면 조심해야 한다.
민감한 원문은 그대로 넣지 않는다.
공개용, 팀원용, 본인용을 처음부터 나눈다.
```

---

## 넣어도 비교적 안전한 자료

```text
공개 블로그 글
공개 포트폴리오
공개 이력서
직접 쓴 공개 가능한 자기소개
일하는 방식에 대한 일반 설명
```

---

## 가명화 후 사용할 자료

```text
회사 self-review 요약
인사평가 요약
동료 피드백 요약
프로젝트 회고
갈등 회고
면접 회고
```

### 처리 방법

```text
회사명 제거
실명 제거
고객명 제거
숫자는 범위로 바꾸기
내부 프로젝트명은 일반명으로 바꾸기
타인의 평가는 삭제하거나 행동 단위로 완화하기
```

---

## 본인용에만 조심해서 쓸 자료

```text
심리검사 결과
상담 메모
건강 관련 내용
가족/관계 이슈
강한 감정 기록
```

이런 자료는 협업자용이나 공개용에는 그대로 넣지 않습니다. 필요하다면 행동 패턴으로 바꿉니다.

```text
원문: 저는 불안이 심하다.
완화: 불확실한 상황에서 정보와 다음 행동이 없으면 빠르게 소진된다.
```

---

## 넣지 말아야 할 자료

```text
비밀번호
API Key
토큰
고객 데이터
계약서 원문
법무/분쟁 자료
타인의 개인정보
회사의 미공개 전략
접근 권한이 제한된 내부 문서 원문
```

---

## 실수로 민감 자료를 올렸다면

이미 민감한 파일이나 원문을 ChatGPT, Project, 다른 LLM 도구에 올렸다면 아래 순서로 정리합니다.

1. 해당 채팅이나 Project에서 파일을 삭제합니다.
2. [자료 목록](../templates/source_register.md)에 삭제 여부와 날짜를 남깁니다.
3. 원문 대신 가명화한 요약본을 새로 작성합니다.
4. 사람 이름, 회사명, 고객명, 고유 숫자, 날짜, 내부 프로젝트명을 제거합니다.
5. 이미 만든 전체 사용 설명서, Active Context, 팀원용 1페이지에 흔적이 남았는지 확인합니다.
6. 남아 있다면 [저자성 검수 프롬프트](../prompts/07_authorship_review.md)로 민감 정보 흔적을 다시 검사합니다.

삭제했다고 해서 도구 제공자의 보관 정책까지 통제할 수 있는 것은 아닙니다. 그래서 민감한 원문은 처음부터 올리지 않는 쪽이 가장 안전합니다.
