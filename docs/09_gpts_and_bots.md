# GPTs와 챗봇으로 확장하기

[← 이전](08_team_sharing.md) | [목차](../SUMMARY.md)

## 이 문서에서 다루는 내용

완성된 사용 설명서를 반복적으로 쓰기 위해 GPTs나 챗봇으로 확장하는 방법을 이해합니다.

처음 사용 설명서를 만드는 중이라면 아직 만들 필요가 없습니다. 먼저 Active Context와 공유용 문서가 실제로 쓸 만한지 확인하세요.

---

## GPTs란?

GPTs는 특정 목적에 맞게 지시문, 지식 파일, 기능을 설정해 사용하는 맞춤형 ChatGPT입니다.

이 가이드에서는 GPTs를 필수로 쓰지 않습니다. 하지만 반복적으로 같은 맥락을 제공해야 한다면 유용할 수 있습니다.

출처 메모: GPTs의 현재 기능, 구성 요소, 공유 범위는 [OpenAI GPTs 문서](../references.md#openai-공식-문서)를 기준으로 확인하세요.

---

## 언제 GPTs로 만들면 좋은가

```text
매번 Active Context를 첨부하기 귀찮다.
개인용 글쓰기 보조 GPT를 만들고 싶다.
피드백 소화, 아이디어 정리, 커리어 문서 작성처럼 반복 작업이 있다.
팀에서 공통 워크북 진행자를 만들고 싶다.
```

---

## 넣으면 좋은 파일

```text
Active Context
답변 취향 로그
팀원용 1페이지
프로젝트 기본 지침의 핵심 규칙
```

전체 사용 설명서를 통째로 넣는 것은 항상 최선이 아닙니다. 너무 길고 민감할 수 있습니다.

---

## 어디에 무엇을 넣나요?

GPTs나 챗봇으로 확장할 때는 파일을 한곳에 다 넣기보다 역할을 나눕니다. 제품 화면 이름은 바뀔 수 있지만, 개념은 아래처럼 맞추면 됩니다.

```text
Instructions:
- 이 GPT가 해야 할 일
- 답변 원칙
- 금지할 행동
- 공개 범위 확인 규칙

Knowledge 또는 참고 파일:
- Active Context
- 답변 취향 로그 중 민감하지 않은 요약
- 팀원용 1페이지

Conversation starters:
- 실제로 자주 묻는 요청 3~5개

Capabilities 또는 tools:
- 웹 검색이 필요한 작업인지
- 파일 분석이 필요한 작업인지
- 이미지나 코드 기능이 필요한지
```

처음부터 GPTs로 만들 필요는 없습니다. Active Context를 일반 채팅에 몇 번 붙여 써보고, 반복해서 쓸 만하다는 확신이 생긴 뒤에 옮깁니다.

출처 메모: Instructions, Knowledge, Conversation starters, Capabilities 같은 항목명과 제공 범위는 제품 업데이트에 따라 달라질 수 있습니다. 최신 상태는 [GPTs 만들기/수정 문서](../references.md#openai-공식-문서)를 확인하세요.

---

## GPTs 지시문 예시

```text
이 GPT는 특정 사용자의 Active Context와 답변 취향 로그를 참고해 답변한다.
가장 화려한 답변보다, 사용자가 납득하고 바로 쓸 수 있는 답변을 만든다.
사용자가 말하지 않은 경험은 만들지 않는다.
필요한 정보가 부족하면 먼저 질문한다.
민감한 내용은 공개용 표현으로 완화한다.
```

---

## 주의

```text
GPTs는 문서를 대신 읽어주는 인터페이스이지, 사용자를 완벽히 아는 존재가 아니다.
민감한 파일은 넣지 않는다.
업데이트한 Active Context를 주기적으로 교체한다.
파일을 교체했으면 예전 파일이 남아 있지 않은지 확인한다.
GPT가 사용자 말을 과하게 다듬으면 답변 취향 로그에 나쁜 예시로 남긴다.
```

출처 메모: GPTs, Memory, Project는 서로 같은 기능이 아닙니다. 특히 GPTs가 저장 메모리나 기존 대화를 어떻게 다루는지는 [GPTs 문서](../references.md#openai-공식-문서), [Memory FAQ](../references.md#openai-공식-문서), [Projects 문서](../references.md#openai-공식-문서)를 나눠서 확인해야 합니다.

## 개인 Project 운영 예시

개인용 Project 하나에 이 워크북의 가이드, 결과물, 원자료를 함께 넣는 방식도 있습니다.

여기서 가이드나 사용 설명서 역할을 하는 파일을 제외하면, 나머지는 거의 전부 원자료입니다. 예를 들면 기존 자기소개서, 이력서, 심리/성향 검사 결과, 상담 기록, 홈페이지 원문, 성장 기록, 오래전에 만든 사용자 설명서 같은 자료입니다.

이런 구성이 유용한 이유는 결과물이 결국 압축본이기 때문입니다. 평소에는 Active Context처럼 짧은 문서만 참고해도 충분하지만, LLM이 더 깊은 근거가 필요하다고 판단할 때는 원자료까지 찾아볼 수 있게 해두면 좋습니다.

대신 이 방식은 보안적으로 더 위험합니다. 민감한 원문이 Project 안에 같이 들어가고, 도구의 파일 보관 정책도 완전히 통제할 수 없기 때문입니다. 그래서 기본값으로 권할 방식은 아닙니다.

이런 Project에는 시스템 지침도 따로 둘 수 있습니다. 그 지침에는 이 사용 설명서를 우선 참고하되, 필요하면 다른 원자료를 찾아보고, 원자료에서 가져온 내용은 확정 사실과 해석을 구분하라고 적어두면 좋습니다. 그러면 GPT가 매번 모든 파일을 다 읽으려 하기보다, 답변에 필요한 파일을 골라 탐색합니다.

출처 메모: Project에 파일과 지침을 함께 두는 방식은 [Projects in ChatGPT](../references.md#openai-공식-문서)의 사용 방식과 닿아 있습니다. 다만 원자료를 함께 넣는 선택은 보안 위험을 키우므로 [File Uploads FAQ](../references.md#openai-공식-문서)와 [보안 참고문헌](../references.md#보안-개인정보-비식별화)을 같이 확인해야 합니다.

이 방식을 쓰려면 최소한 아래는 지키는 편이 좋습니다.

```text
Project 안의 파일을 가이드/결과물/원자료로 구분한다.
원자료를 넣는 이유와 감수하는 위험을 스스로 알고 있어야 한다.
회사 기밀, 타인 개인정보, 고객 데이터, 접근 제한 문서는 넣지 않는다.
원자료에서 가져온 말은 그대로 자기 설명으로 확정하지 않는다.
민감한 내용은 팀원용/공개용 문서로 옮기기 전에 다시 검수한다.
```
