# 프롬프트 06. 최종 사용 설명서 생성

[← 이전](05_round2_interview.md) | [목차](../SUMMARY.md) | [다음 →](07_authorship_review.md)

## 언제 쓰나요?

모든 원자료를 모은 뒤, 최종 Markdown 파일들을 생성할 때 사용합니다.

## 필요한 입력

`source_register.md`, `01_answer_log.md`, `02_round1_summary.md`, `03_followup_questions.md`, `04_round2_summary.md`, 선택 자료 요약.

## 기대 결과

`full_manual.md`, `active_context.md`, `team_share_one_pager.md`, `public_profile.md`, `update_plan.md`가 생성됩니다.

---

```text
아래 자료들은 한 사람의 개인 사용 설명서를 만들기 위한 원자료다.

이 작업의 목표는 가장 화려하고 완성도 높아 보이는 글을 만드는 것이 아니다.
목표는 이 사람이 “이건 내 맥락과 맞다”, “내가 쓴 것처럼 느껴진다”, “수정 없이 쓸 수 있다”고 납득할 수 있는 문서를 만드는 것이다.

중요한 작성 원칙:
1. 사용자가 직접 말한 내용과 해석을 분리한다.
2. 사용자가 말하지 않은 경험, 동기, 과거, 감정을 지어내지 않는다.
3. 확실한 내용, 가설, 추가 확인 필요를 구분한다.
4. 심리검사나 성향검사는 참고자료일 뿐 정답으로 쓰지 않는다.
5. 진단명이나 성격 낙인 대신 관찰 가능한 행동 패턴으로 쓴다.
6. 너무 멋있게 업그레이드하지 않는다.
7. 사용자의 원래 말투와 사고 순서를 가능한 한 보존한다.
8. 공개 범위를 본인용/팀원용/공개용으로 구분한다.
9. 회사 기밀, 타인 개인정보, 고객 정보, 민감한 원문은 포함하지 않는다.
10. 모르는 내용은 “추가 확인 필요”로 남긴다.

입력 자료:
[여기에 파일을 첨부하거나 요약을 붙여넣기]

출력은 채팅 본문 하나로 길게 쓰지 말고, 아래 Markdown 파일 단위로 나누어 작성한다.

---

# 파일 1: outputs/05_full_manual.md

## 0. 이 문서의 사용법
- 이 문서가 무엇인지
- 무엇이 아닌지
- 본인용/팀원용/공개용 주의
- 업데이트 전제

## 1. 한 문장 설명
- 이 사람을 한 문장으로 설명
- 단, 과장된 브랜딩 문장으로 만들지 말 것
- 사용자가 실제로 받아들일 수 있는 표현으로 작성

## 2. 현재의 자기 인식
다음 항목을 포함:
- 사용자가 스스로를 설명한 말
- 타인이 오해하기 쉬운 점
- 오래 본 사람이 알게 되는 점
- 현재 확실한 부분
- 아직 유동적인 부분

## 3. 살아온 맥락과 반복 주제
다음 항목을 포함:
- 중요한 전환점
- 반복되는 선택 패턴
- 예전과 지금 달라진 점
- 계속 등장하는 키워드
- 단, 사적인 내용은 공개 범위 표시

## 4. 코어 드라이브와 가치관
다음 항목을 포함:
- 무엇이 이 사람을 움직이는가
- 의미 있다고 느끼는 조건
- 무의미하다고 느끼는 조건
- 존중받는다고 느끼는 장면
- 무시당한다고 느끼는 장면
- 타협 가능/불가능한 기준

## 5. 직관과 판단 방식
다음 항목을 포함:
- 이 사람이 빠르게 감지하는 신호
- 직관이 맞았던 사례
- 직관이 틀렸던 사례
- 판단 전에 필요한 정보
- 다른 사람에게 설명하기 어려운 이유
- 직관을 설득 가능한 말로 바꾸는 방법

## 6. 일하는 방식
다음 항목을 포함:
- 잘하는 일
- 오래 하고 싶지 않은 일
- 자연스럽게 맡는 역할
- 좋은 업무 요청의 조건
- 나쁜 업무 요청의 조건
- 자율성과 책임의 균형
- 문제 해결 방식
- 강점이 과해질 때의 부작용

## 7. 커뮤니케이션 인터페이스
다음 항목을 포함:
- 잘 이해하는 설명 방식
- 혼란스러워지는 설명 방식
- 말이 길어지는 이유
- 말하지 않고 삼키는 생각
- 피드백을 받을 때 필요한 정보
- 칭찬이 진짜라고 느껴지는 조건
- 갈등이 생겼을 때 원하는 첫 대화

## 8. 에너지와 회복
다음 항목을 포함:
- 에너지가 올라가는 조건
- 에너지가 떨어지는 조건
- 집중이 잘 되는 조건
- 집중이 무너지는 조건
- 지치기 시작할 때 첫 신호
- 많이 소진됐을 때의 행동
- 진짜 휴식과 가짜 휴식
- 최소 유지 조건

## 9. 관계와 경계
다음 항목을 포함:
- 편안한 사람의 특징
- 긴장되는 사람의 특징
- 신뢰가 생기는 조건
- 신뢰가 깨지는 조건
- 도움 요청 방식
- 혼자 있고 싶은 것과 고립의 차이
- 공개 가능한 영역과 비공개 영역

## 10. LLM 답변 취향
이 장은 반드시 자세히 쓴다.
다음 항목을 포함:
- 바로 쓸 수 있다고 느끼는 답변의 특징
- 그럴듯하지만 내 것이 아니라고 느끼는 답변의 특징
- 선호하는 답변 구조
- 싫어하는 말투
- 원하는 디테일 수준
- 과한 업그레이드의 예
- 한 줄에서 긴 글로 확장하는 방식
- 답변 전에 먼저 물어봐야 하는 질문

## 11. 나와 일하는 법
팀원이나 협업자가 읽고 행동을 바꿀 수 있게 쓴다.
다음 항목을 포함:
- 좋은 요청 예시
- 나쁜 요청 예시
- 피드백 예시
- 갈등 시 확인 질문
- 하지 말아야 할 행동
- 도움이 되는 행동

## 12. 위기와 리스크
진단하지 말고 행동 패턴으로 쓴다.
각 항목마다:
- 신호
- 하면 안 되는 대응
- 도움이 되는 대응
- 24시간 대응
- 예방 루틴
다룰 수 있는 항목:
- 번아웃
- 자기비판 루프
- 영향력 부재로 인한 동기 저하
- 과한 완벽주의
- 관계 회피 또는 고립
- 답변을 자기합리화로 쓰는 위험

## 13. 성장 방향
다음 기간으로 나누어 작성:
- 3개월
- 6개월
- 1년
- 3년
각 기간에 대해:
- 일/커리어
- 관계/협업
- 건강/회복
- 학습/창작

## 14. Quick Command Palette
실제로 GPT에게 넣을 수 있는 명령어를 만든다.
예:
- /context_first
- /ask_me_first
- /one_line_then_expand
- /explain_my_intuition
- /feedback_digest
- /team_conflict_bridge
- /recovery_mode
- /public_version
각 명령어마다 목적과 예시를 포함한다.

## 15. 추가 확인 필요
모르는 것을 그럴듯하게 채우지 말고 목록으로 남긴다.
```
