# 품질 테스트 질문

[목차](../SUMMARY.md)

## 언제 쓰면 좋나요?

최종 사용 설명서나 active_context를 업데이트한 뒤, 실제로 답변 품질이 좋아졌는지 확인할 때 사용합니다.

---

## 테스트 방법

1. 같은 모델로 새 채팅을 2개 엽니다.
2. 첫 번째 채팅에서는 `active_context.md` 없이 아래 질문 중 1~3개를 실행합니다.
3. 두 번째 채팅에서는 이전 답변을 보여주지 말고, 같은 질문을 `active_context.md`와 함께 실행합니다.
4. 두 답변을 나란히 비교합니다.
5. 점수 차이가 난 이유를 `answer_taste_log.md`에 남깁니다.
6. 반복해서 보이는 차이만 `active_context.md`에 반영합니다.

정확한 정량 평가는 어렵습니다. 이 테스트는 점수를 매기려는 절차가 아니라, 답변이 반복해서 나아지는지 보려는 샘플 질문입니다. 생각나는 차이가 있으면 `answer_taste_log.md`에 좋은 예시나 나쁜 예시로 남깁니다.

---

## 테스트 질문

```text
1. 제가 어떤 아이디어를 직관적으로 떠올렸는데 이유를 설명하기 어렵다고 하자. 그 생각을 설득 가능한 방식으로 정리해줘.

2. 아래 한 줄 아이디어를 제 말투에 맞게 10줄로 확장해줘. 너무 멋있게 만들지 말고 자연스럽게 써줘.

3. 제가 팀원과 의견이 안 맞는 상황이다. 누가 맞는지 판단하지 말고, 서로 놓친 맥락을 확인할 질문을 만들어줘.

4. 제가 받은 피드백을 사실, 감정, 추론, 다음 행동으로 나눠줘.

5. 아래 문장을 공개용 블로그 문장으로 바꿔줘. 민감한 내용은 제거하고 일하는 방식 중심으로 바꿔줘.
```

---

## 평가 기준

각 항목은 1~5점으로 기록합니다.

| 항목 | active_context 없음 | active_context 있음 | 메모 |
|---|---:|---:|---|
| 바로 사용 가능성 |  |  | 수정 없이 쓸 수 있었나 |
| 말투 보존 |  |  | 제 말투와 결이 남아 있었나 |
| 근거 충실도 |  |  | 말하지 않은 내용을 만들지 않았나 |
| 구조 적합도 |  |  | 원하는 순서와 깊이에 가까웠나 |
| 질문 적절성 |  |  | 부족한 정보가 있을 때 먼저 물었나 |
| 과한 업그레이드 방지 |  |  | 필요 이상으로 멋있게 만들지 않았나 |

재요청 횟수와 수정 시간도 같이 남깁니다.

```text
재요청 횟수:
수정 시간:
점수가 1점 이상 좋아진 항목:
오히려 나빠진 항목:
active_context에 반영할 조건:
```

## 정확성 확인 질문

답변이 마음에 들어도 자기합리화가 섞였는지 확인합니다.

```text
아래 답변이 저를 기분 좋게 만들기 위해 사실과 해석을 섞은 부분이 있는지 찾아줘.
제 입장에 유리한 해석과 불리한 해석을 분리해줘.
검증 가능한 사실, 추론, 감정, 선택지를 나눠줘.
```

## 기록하면 좋은 것

```text
바로 쓸 수 있었던 답변:
제가 다시 고쳐야 했던 표현:
근거 없이 저를 해석한 문장:
너무 멋있게 바뀐 문장:
active_context 없이 더 나았던 부분:
active_context를 붙였을 때 좋아진 부분:
다음 active_context에 추가할 조건:
평가가 어려워 보류한 항목:
```
